LAN-Share Project Plan

An interactive guide to your P2P file transfer application.

Project Roadmap

This roadmap breaks the project into three manageable phases. Start with Phase 1 to build the core functionality and progress from there.

Phase 1: Core CLI Tool

1.1: Project Setup

Create a Git repo, set up `CMakeLists.txt` for C++20, and write a "Hello, World!" to verify the toolchain.

1.2: Networking Foundation

Set up Boost.Asio. Create a simple TCP echo server & client to ensure basic networking is solid.

1.3: File Sending Logic

Define a transfer protocol (header with filename/size). Implement client logic to connect, send header, and stream file chunks.

1.4: File Receiving Logic

Implement server logic to listen, read the header, create the file, and write incoming chunks to disk.

1.5: Build the CLI

Integrate sender/receiver logic into one executable. Use command-line arguments to select modes and parameters.

Phase 2: Web Interface

2.1: Integrate HTTP Library

Research and integrate a simple C++ HTTP library (e.g., `cpp-httplib`) into the CMake project.

2.2: Create Web Module

Create a `WebServer` class running in its own thread. Set up handlers for `/` (HTML page) and `/upload` (POST requests).

2.3: Build the Web Page

Create a simple `index.html` with a file input form that POSTs to the `/upload` endpoint.

2.4: Connect Web to Core

When the `/upload` handler receives a file, pass its data to the Core Logic's Transfer Manager to reuse existing logic.

Phase 3: Extended Features

3.1: Network Discovery

Research UDP Broadcast/Multicast. Implement a discovery mechanism for clients to find each other on the LAN.

3.2: TUI Interface

Research and integrate a TUI library (e.g., FTXUI, ncurses) to create a more interactive terminal experience.

Future: Compression

Integrate a library like zlib to add optional file compression before sending and decompression on receipt.

Future: Encryption

Add optional end-to-end encryption for file data during transfer using a library like OpenSSL.

Module Descriptions

  • User Interfaces (UI)

    The user-facing parts of the application. These modules are responsible for presenting information and capturing user input, but contain no core transfer logic. They communicate with the Core Logic via a clean API.

  • Core Logic

    The brain of the application. The **Transfer Manager** orchestrates file transfers. It uses the **Network Protocol** to define communication, **File I/O** to handle disk access, and future modules for discovery and compression.

  • Low-Level Services

    The foundation providing low-level capabilities. The **Networking** service, built with a library like Boost.Asio, is responsible for all TCP/UDP communication, sending and receiving raw bytes over the network.

Core Concepts

Traditional FTP uses two connections (control and data) and is a rigid client-server protocol. We are building a simpler, more direct peer-to-peer system over a single TCP socket with our own custom protocol. This is more lightweight and better for learning direct network communication.

We use TCP sockets for reliable data streams. The process is: 1) Sender connects to Receiver. 2) Sender sends a header with metadata (filename, size). 3) Sender reads the file in chunks and sends them. 4) Receiver reads chunks and writes them to a new file until the full size is received.

  • RAII: Use smart pointers (`std::unique_ptr`) to manage resources automatically.
  • `std::filesystem`: Use this for cross-platform file and path manipulation.
  • Asynchronous I/O: Use Boost.Asio to avoid "blocking" and keep the application responsive.
  • Modular Design: Separate networking, file handling, and UI code as shown in the architecture.
